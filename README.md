# 커피숍 주문 시스템

## 1. 개요

이 프로젝트는 포인트 기반으로 동작하는 커피숍 주문 시스템을 구현한다.

사용자는 커피 메뉴를 조회할 수 있고, 포인트를 충전한 뒤 해당 포인트로 메뉴를 주문한다.
또한 주문 내역은 외부 데이터 수집 플랫폼으로 실시간 전송되어야 하며, 최근 7일간 주문 데이터를 바탕으로 인기 메뉴를 조회할 수 있다.

## 2. 요구사항 분석

1. 커피 메뉴 목록 조회
    - 메뉴 ID, 메뉴명, 가격 정보를 조회할 수 있어야 한다.

2. 포인트 충전
    - 사용자는 포인트를 충전할 수 있어야 한다.
    - 결제 수단은 포인트만 사용 가능하므로, 주문 이전에 포인트 충전 기능이 필요하다.

3. 커피 주문 / 결제
    - 사용자는 메뉴를 선택해 주문할 수 있어야 한다.
    - 주문 시 메뉴 가격만큼 포인트가 차감되어야 한다.
    - 주문 완료 내역은 외부 데이터 수집 플랫폼으로 실시간 전송되어야 한다.

4. 인기 메뉴 조회
    - 최근 7일간 주문 데이터를 기준으로 인기 메뉴 3개를 조회할 수 있어야 한다.
    - 메뉴별 주문 횟수 집계는 정확해야 한다.

- 애플리케이션이 다수 서버 / 다수 인스턴스 환경에서도 문제없이 동작해야 한다.
- 포인트 충전 및 주문 시 동시성 이슈를 고려해야 한다.
- 주문, 결제, 외부 전송 과정에서 데이터 일관성을 고려해야 한다.
- 각 기능과 제약사항에 대한 테스트가 필요하다.

## 3. 문제 해결 전략

이번 프로젝트는 처음부터 모든 복잡도를 한 번에 반영하는 것 보다 필수 요구사항을 먼저 만족한 이후에 점진적으로 확장하는 방식으로 진행한다.

우선 메뉴 조회, 포인트 충전, 주문 / 결제, 인기 메뉴 조회의 필수 기능을 중심으로 구현한다.
이 과정에서 도메인을 단순하게 유지하면서 이후 동시성 제어와 데이터 일관성 보완이 가능하도록 구조를 잡는다.

특히 주문 / 결제 기능은 포인트 차감이 함께 일어나기 때문에 단순 조회 기능과 다르게 정합성이 중요하다.
따라서 주문과 포인트 차감은 하나의 흐름으로 관리할 수 있도록 설계하고, 인기 메뉴 집계는 최근 7일간의 주문 완료 데이터를 기준으로 조회하는 방향으로 시작한다.

외부 데이터 수집 플랫폼 전송 역시 먼저 요구사항을 만족하는 방식으로 구현하고,
이후 단계에서 실패 처리, 재시도, 트랜잭션과의 관계를 추가로 검토한다.

## 8. 동시성 / 데이터 일관성

가장 중요한 정합성 대상은 사용자의 포인트 잔액과 주문 데이터이다.

포인트는 충전과 차감이 모두 발생하는 상태 데이터이고 동시에 여러 요청이 들어올 경우 잔액이 잘못 계산될 수 있다.
특히 주문 / 결제 기능에서는 포인트 차감과 주문 생성이 함께 일어나기 때문에 일부만 반영되는 상황이 발생하지 않도록 설계가 필요하다.

따라서 주문 / 결제 과정에서는 다음 흐름을 하나의 논리 단위로 묶었다.

- 포인트 차감
- 주문 생성
- 포인트 사용 이력 저장

위의 세 단계는 함께 성공하거나 실패해야 하고 하나의 트랜잭션으로 관리해야한다.

반면, 주문 내역의 외부 데이터 수집 플랫폼 전송은 네트워크나 외부 시스템 상태에 영향을 받을 수 있기 때문에
결제 흐름과는 분리한다.

또한 인기 메뉴 조회는 최근 7일간의 주문 완료 데이터를 기준으로 집계하기 때문에
정확한 주문 기록이 필요하며 캐시나 애플리케이션 메모리 보다 데이터베이스의 주문 데이터를 기준으로 처리해야 한다.
